# Outlook OTP 轮询诊断日志设计

## 背景
ChatGPT 注册链路在发送邮箱验证码后，会进入 Outlook 本地池邮箱的阻塞式收码阶段。

当前现象是：
- 日志只显示“正在等待邮箱验证码”
- 实际内部会持续轮询 IMAP
- 但轮询过程没有中间诊断日志
- IMAP 查询异常也会被静默吞掉，外部看起来像“卡住”

这导致无法快速区分以下情况：
1. 邮件根本没收到
2. 邮件进入 Junk 而不是 INBOX
3. IMAP 登录或查询异常
4. 邮件收到了，但验证码提取失败

## 目标
在不改变现有收码行为的前提下，为 Outlook 本地池收码流程补充最小但足够的诊断日志，帮助快速定位 OTP 阶段阻塞原因。

## 方案
采用最小范围的实现：只修改 Outlook 收码逻辑，不改 ChatGPT 主链路，不改邮箱选择逻辑。

改动位置：
- `core/base_mailbox.py`
- 重点函数：`OutlookMailbox.wait_for_code()`

### 日志范围
本次新增日志包括：

1. **每轮轮询概览**
   - 当前轮询的文件夹
   - IMAP 登录是否成功
   - 本轮 UID 总数
   - 新增 UID 数量

2. **命中邮件时的精简日志**
   - 命中的邮件 subject
   - 是否提取到验证码
   - 若验证码命中但属于 `exclude_codes`，打印“跳过已尝试验证码”

3. **异常日志**
   - IMAP 登录失败原因
   - select/search/fetch 失败原因
   - 其他轮询异常原因

4. **双文件夹诊断**
   - 在 `INBOX` 与 `Junk` 两个文件夹上分别执行相同轮询逻辑
   - 日志中明确标出当前文件夹
   - 先查 `INBOX`，再查 `Junk`

## 设计细节
### 保持行为不变
- 仍然沿用现有超时机制与 5 秒轮询间隔
- 仍然只在提取到可用验证码时返回
- 仍然支持 `before_ids` 与 `exclude_codes`
- 不修改上层 ChatGPT 注册流程

### 日志控制原则
为避免日志过脏：
- 不打印邮件正文
- 不打印全部邮件列表
- 不打印完整 token / 凭证
- 只打印统计信息、subject、提取结果、异常原因

### 文件夹策略
本次仅增加最常见的第二诊断文件夹：`Junk`。

实现上会：
- 对每轮 poll，依次尝试 `INBOX`、`Junk`
- 每个文件夹独立打印 select/search 结果
- 共享同一组 `seen` / `exclude_codes` 过滤逻辑

这样可以定位：
- 邮件是否只进垃圾箱
- 是否两边都没有新邮件
- 是否只有某个文件夹查询失败

## 备选方案对比
### 方案 A：仅在 OutlookMailbox.wait_for_code 打精简日志，并增加 INBOX/Junk 双文件夹诊断（推荐）
- 优点：改动最小，定位信息最直接，贴近真实轮询点
- 优点：能明确区分 INBOX 与 Junk 的结果
- 缺点：OutlookMailbox 内部日志逻辑略有增加

### 方案 B：在 BaseMailbox._run_polling_wait 打通用日志
- 优点：可复用于其他邮箱实现
- 缺点：拿不到 Outlook 特有信息，如 UID 数量、subject、文件夹差异
- 缺点：对本次问题帮助有限

## 影响范围
- 修改文件：`core/base_mailbox.py`
- 不涉及：ChatGPT 主状态机、邮件导入逻辑、前端页面、数据库结构

## 验证方式
完成后应能在实际日志中看到：
1. 进入 OTP 等待后，出现周期性的 Outlook 轮询日志
2. 能区分 `INBOX` 与 `Junk` 的查询结果
3. 若 IMAP 异常，日志中能看到异常原因
4. 若邮件到达但提取不到验证码，至少能看到命中的邮件 subject
5. 若验证码被 `exclude_codes` 过滤，日志中能看到跳过提示

## 风险
风险较低。

本次只增加诊断日志与 `Junk` 文件夹轮询，不修改现有业务决策与超时行为。

主要注意点：
- 日志不能泄露敏感正文
- 不要因为日志分支改变原有返回条件
- 双文件夹轮询不能破坏现有 `seen` 去重逻辑
